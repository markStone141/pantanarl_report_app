# DairyMetrics 仕様メモ

このファイルは、将来的に `DairyMetrics` を実装する際の叩き台です。
現時点では構想段階のため、実装前提の確定仕様ではなく、判断軸と設計方針を整理するためのメモとして扱います。

## 1. 仮称

- アプリ名（仮）: `DairyMetrics`
- 位置づけ:
  - 現在の `Report App` を置き換えるのではなく、日次実績入力を支える新しい入力導線
  - 既存の代表者による最終報告フローは維持する

## 2. 背景

- 現在は Excel の「毎日実績集計表」で日々の実績を管理している
- Excel はスマホ入力しづらく、ランキングや平均も見づらい
- メンバーは活動後に疲れており、入力忘れが起きやすい
- 代表者による最終報告スタイルは残したい
- 入力済みのメンバー実績は、代表者の報告フォーム側に自動反映したい

## 3. プロダクト目標

- Excel 入力を廃止し、Web アプリだけで日次入力を完結させる
- メンバーごとに `ログイン -> 当日実績入力` を最短導線で完了できるようにする
- モバイルでの入力負荷を極小化する
- 毎日見たくなる軽さと、少し前向きになれる UI を作る
- 入力したデータから平均、ランキング、進捗などを可視化する
- 代表者向け報告フォームには、既存の運用を維持しつつ、入力済み実績を初期値反映する

## 4. 対象ユーザー

- メンバー
  - 自分の当日実績を入力する
  - 入力は主にスマホを想定
- 代表者
  - メンバー入力済みデータを確認する
  - 必要に応じて補正し、最終的な日報を提出する
- 管理者
  - メンバー、部署、ランキング、目標、履歴を確認する

## 5. 基本フロー

### 5.1 メンバー

1. ID / パスワードでログイン
2. 当日分の自分の入力画面へ直接到達
3. 実績を入力して保存
4. 保存後に、その日の進捗や簡易ランキングを見られる

### 5.2 代表者

1. 既存の代表者向け報告フォームを開く
2. 対象日のメンバー入力済み実績があれば、初期値として反映される
3. 必要に応じて調整
4. 最終提出

## 6. 入力体験の方針

- モバイルファースト
- ログイン後すぐ入力できる
- 1画面1目的を優先する
- 数字入力を中心にして、迷う項目を減らす
- 当日入力の保存状態が明確に分かる
- 「未入力」「入力済み」「報告反映済み」が分かる
- 長い表形式を避け、カード型またはステップ型を優先する

## 7. Excel から読み取れる入力要素

Excel の月別シートでは、メンバーごとに日次入力ブロックがある。
現時点で読み取れた主な入力要素は以下。

- メンバー名
- アプローチ
- コミュニケーション
- 会員
- 支援金
- 現場名
- 月目標
- 実績
- 残り

ただし、実運用上どこまでをメンバー自身に入力させるかは再設計が必要。
特に `月目標` や `残り` は入力項目ではなく、集計結果として算出する方が自然。

## 8. 推奨データ設計案

### 8.1 新規の個人入力モデルを持つ

既存 `DailyDepartmentReport` に直接メンバーの途中入力を入れるのではなく、
「個人の日次下書き」相当のモデルを持つ方が安全。

想定モデル名:

- `MemberDailyMetricEntry`

候補フィールド:

- `member`
- `department`
- `entry_date`
- `approach_count`
- `communication_count`
- `member_count`
- `support_amount`
- `location_name`
- `memo`
- `status`
  - `draft`
  - `submitted`
  - `synced_to_report`
- `submitted_at`
- `updated_at`

### 8.2 代表者報告との関係

- 代表者の報告フォーム表示時に、対象日・対象部署の `MemberDailyMetricEntry` を読む
- 既存の `DailyDepartmentReportLine` に変換できるデータは初期値として反映する
- 代表者が最終提出したら、該当入力を `synced_to_report` に更新する

## 9. 既存 Report App との接続方針

- 既存の `Report App` は最終報告の器として残す
- `DairyMetrics` はその手前の「日次入力レイヤー」とする
- 実装順は以下が安全

1. メンバー個人入力の保存
2. 代表者フォームへの初期値反映
3. 同期済み状態の表示
4. ランキング・平均・進捗表示

## 10. 画面案

### 10.1 メンバー向け

- ログイン画面
  - 軽い
  - 毎日開きたくなる印象
- 今日の入力画面
  - 自分の名前
  - 今日の日付
  - 数字入力中心
  - 保存ボタン
  - 前回入力との差分確認があると便利
- マイ実績画面
  - 今月累計
  - 平均
  - 過去7日

### 10.2 代表者向け

- 既存報告フォームに「メンバー入力済み」表示を追加
- 反映済み件数、未入力件数を表示
- 取り込み元データを上書きできるようにする

### 10.3 管理者向け

- ランキング
- 月間平均
- 路程ごとの進捗
- 未入力者一覧
- 入力率ダッシュボード

## 11. 可視化で重視するもの

- 個人ランキング
- 月間累計
- 平均値
- 目標との差分
- 路程進捗
- 部署別比較
- 連続入力日数

特に「モチベーション向上」に寄せるなら、単なる一覧より以下が有効。

- 自分の現在順位
- 前日比
- 今月の自己ベスト
- チーム内平均との差

## 12. UX 方針

- 義務感の強い業務画面にはしない
- 速度を重視する
- 操作数を減らす
- 入力完了後に少し達成感がある
- ランキングや進捗は責める印象ではなく、振り返りしやすい見せ方にする

## 13. 段階導入案

### Phase 1

- メンバー個人ログイン
- 当日実績入力
- 保存

### Phase 2

- 代表者フォームへの自動反映
- 入力済み / 未入力 状態表示

### Phase 3

- ランキング
- 平均
- 目標との差分
- 月間サマリー

### Phase 4

- 通知
- 入力忘れリマインド
- 連続入力や達成バッジなどの軽いゲーミフィケーション

## 14. 実装判断メモ

- Excel レイアウトの完全再現は目指さない
- Excel 由来の項目は尊重するが、UI は Web / モバイル向けに再設計する
- メンバー入力と代表者報告を同じモデルに無理に載せない
- 途中入力と最終報告は責務を分ける
- 集計系は保存値より算出値を優先する
