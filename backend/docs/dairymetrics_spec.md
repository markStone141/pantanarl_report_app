# DairyMetrics 仕様メモ

このファイルは、将来的に `DairyMetrics` を実装する際の叩き台です。
現時点では構想段階のため、実装前提の確定仕様ではなく、判断軸と設計方針を整理するためのメモとして扱います。

## 1. 仮称

- アプリ名（仮）: `DairyMetrics`
- 位置づけ:
  - 現在の `Report App` を置き換えるのではなく、日次実績入力を支える新しい入力導線
  - 既存の代表者による最終報告フローは維持する

### 現時点の対象範囲

- まずは `UN` と `WV` を対象に実装する
- `Style` 系は初期対象外とし、後から展開する

## 2. 背景

- 現在は Excel の「毎日実績集計表」で日々の実績を管理している
- Excel はスマホ入力しづらく、ランキングや平均も見づらい
- メンバーは活動後に疲れており、入力忘れが起きやすい
- 代表者による最終報告スタイルは残したい
- 入力済みのメンバー実績は、代表者の報告フォーム側に自動反映したい

## 3. プロダクト目標

### 3.1 主目的

- 毎日の実績を無理なく入力し、その蓄積データを継続的に見返せるようにする
- 蓄積されたデータを可視化し、他者比較、自己振り返り、モチベーション向上に役立てる
- 平均、ランキング、目標差分などを通じて、新しい戦略や改善行動を立てやすくする

### 3.2 実現手段

- Excel 入力を廃止し、Web アプリだけで日次入力を完結させる
- メンバーごとに `ログイン -> 当日実績入力` を最短導線で完了できるようにする
- モバイルでの入力負荷を極小化する
- 毎日見たくなる軽さと、少し前向きになれる UI を作る
- 入力したデータから平均、ランキング、進捗などを可視化する

### 3.3 副次効果

- 代表者向け報告フォームには、既存の運用を維持しつつ、入力済み実績を初期値反映する
- 結果として、代表者による日報作成や確認の手間を減らす

## 4. 対象ユーザー

- メンバー
  - 自分の当日実績を入力する
  - 入力は主にスマホを想定
- 代表者
  - メンバー入力済みデータを確認する
  - 必要に応じて補正し、最終的な日報を提出する
- 管理者
  - メンバー、部署、ランキング、目標、履歴を確認する
  - 後追い実績を追加・補正する

## 5. 基本フロー

### 5.1 メンバー

1. ID / パスワードでログイン
2. メンバーダッシュボードへ到達
3. ダッシュボードから当日分の実績入力へ進む
4. 実績を入力して保存
5. 保存後にダッシュボードへ戻り、自分の進捗や成績を見られる

### 5.2 代表者

1. 既存の代表者向け報告フォームを開く
2. 対象日のメンバー入力済み実績があれば、初期値として反映される
3. 必要に応じて調整
4. 最終提出

### 5.3 管理者

1. 月単位で全体実績を確認する
2. 郵送申込や増額など、後から発生した実績を追加する
3. 必要に応じて個人入力や最終報告に反映させる

## 6. 入力体験の方針

- モバイルファースト
- `ログイン -> ダッシュボード -> 実績入力` の流れを短く保つ
- 1画面1目的を優先する
- 数字入力を中心にして、迷う項目を減らす
- 当日入力の保存状態が明確に分かる
- 「未入力」「入力済み」「報告反映済み」が分かる
- 長い表形式を避け、カード型またはステップ型を優先する

### 6.1 画面設計の前提

- メンバー利用端末は 9 割以上がスマホ想定として設計する
- メンバー画面は `Talks` / `Testimony` のようなモバイル前提 UI を基準にする
- ハンバーガーメニュー、ドロワー NAV、カードレイアウトを初期設計から前提に含める
- 「PC でも使える」ではなく「スマホで気持ちよく使える」を先に満たす
- 管理画面は PC 操作を主軸にするが、モバイルでも破綻しない構成にする
- 管理画面も共通のドロワー NAV ロジックを使い、最低限のモバイル操作性は確保する

## 7. Excel から読み取れる入力要素

Excel の月別シートでは、メンバーごとに日次入力ブロックがある。
現時点で読み取れた主な入力要素は以下。

- メンバー名
- アプローチ
- コミュニケーション
- 会員
- 支援金
- 現場名
- 月目標
- 実績
- 残り

ただし、実運用上どこまでをメンバー自身に入力させるかは再設計が必要。
特に `月目標` や `残り` は入力項目ではなく、集計結果として算出する方が自然。

## 8. 推奨データ設計案

### 8.1 新規の個人入力モデルを持つ

既存 `DailyDepartmentReport` に直接メンバーの途中入力を入れるのではなく、
「個人の日次下書き」相当のモデルを持つ方が安全。

想定モデル名:

- `MemberDailyMetricEntry`

候補フィールド:

- `member`
- `department`
- `entry_date`
- `approach_count`
- `communication_count`
- `member_count`
- `support_amount`
- `location_name`
- `memo`
- `status`
  - `draft`
  - `submitted`
  - `synced_to_report`
- `submitted_at`
- `updated_at`

初期対象部署を踏まえた入力項目メモ:

- `UN`
  - `approach_count`
  - `communication_count`
  - `member_count`
  - `support_amount`
  - `location_name`
- `WV`
  - `approach_count`
  - `communication_count`
  - `member_count` または既存 `WV` の報告指標に合わせた件数項目
  - `support_amount`
  - `location_name`

`WV` の件数項目が現行 `Report App` の `CS / 難民` とどう対応するかは別途マッピング整理が必要。

### 8.2 代表者報告との関係

- 代表者の報告フォーム表示時に、対象日・対象部署の `MemberDailyMetricEntry` を読む
- 既存の `DailyDepartmentReportLine` に変換できるデータは初期値として反映する
- 代表者が最終提出したら、該当入力を `synced_to_report` に更新する

### 8.3 管理者による後追い実績

後から追加される実績（郵送、増額、後日確定分など）は、メンバー本人入力とは別経路で管理者が登録できるようにする。

補足:

- `郵送` は独立カテゴリではなく、`戻り` 実績の一部として扱う
- `戻り` の追加区分は最低限以下の 2 種類を持つ
  - `郵送`
  - `QR`
- 後追い入力時に管理者が最低限入れる値は以下
  - 対象日
  - メンバー
  - 部署
  - 区分
    - `戻り(郵送)`
    - `戻り(QR)`
    - `増額`
    - `その他`
  - 金額
  - 必要なら件数
  - 補足 / 理由

想定追加項目:

- `source_type`
  - `member_input`
  - `admin_adjustment`
  - `return_postal`
  - `return_qr`
  - `increase`
- `adjustment_reason`
- `created_by`
- `applied_at`

設計方針:

- メンバー入力と管理者追加は履歴上区別できるようにする
- 管理者追加分を集計対象に含める
- どの時点で追加されたかを追えるようにする
- `戻り` と `増額` は、月次 / 路程 / 個人ダッシュボードの集計値にも含める
- 管理者追加後に、対象月の全体実績と目標差分がずれないことを優先する
- 既に存在する `Report App` の当日報告と、管理者後追い分は分けて保持しつつ、月次集計上は合算して見られるようにする
- 必要に応じて、対象月・対象日ベースで過去報告の表示値にも反映できるようにする
- 管理者追加は「元の報告を破壊的に書き換える」のではなく、「後追い補正として上乗せ」する方が安全

## 9. 既存 Report App との接続方針

- 既存の `Report App` は最終報告の器として残す
- `DairyMetrics` はその手前の「日次入力レイヤー」とする
- 実装順は以下が安全

1. メンバー個人入力の保存
2. 代表者フォームへの初期値反映
3. 同期済み状態の表示
4. ランキング・平均・進捗表示

## 11. 可視化で重視するもの

- 個人ランキング
- 月間累計
- 平均値
- 目標との差分
- 路程進捗
- 部署別比較
- 先週比 / 先月比
- 成長率

可視化方針:

- 数値だけで直感的に伝わる項目は、まず数値を主役にする
- 推移、比較、増減、順位変化のように、視覚化した方が早く理解できる項目はグラフを積極的に使う
- グラフは飾りではなく、行動や振り返りにつながる情報を短時間で把握するために使う
- モバイルでは複雑な多系列グラフより、単純で読み取りやすいミニグラフや要約グラフを優先する
- 管理画面では PC 前提で、月次俯瞰に向いた比較グラフや推移グラフも活用する

特に「モチベーション向上」に寄せるなら、単なる一覧より以下が有効。

- 自分の現在順位
- 前日比
- 先週比
- 先月比
- 成長率
- 伸びた項目
- 落ちた項目
- 今月の自己ベスト
- チーム内平均との差
- 指標ごとのランキング
- 最高値だけでなく平均値ベースのランキング

グラフ候補:

- 日別推移の折れ線グラフ
- 月目標に対する進捗バー
- 指標ごとの前月比 / 前週比の差分グラフ
- 自分とチーム平均の比較バー
- ランキング上位の簡易比較チャート

メンバーダッシュボードの初期表示候補:

- 個人の月間成績
- 月間の個人目標
- 現路程の個人目標
- 今日の目標
- 累計件数
- 累計支援金額
- アプローチの一日平均
- アプローチの累計
- コミュニケーションの一日平均
- コミュニケーションの累計
- 先週比 / 先月比
- 伸びた項目
- 落ちた項目
- 成長率サマリー

目標表示の方針メモ:

- `月間の個人目標`
  - 個人ごとの月間目標値
  - 現在実績との差分
- `現路程の個人目標`
  - 現在の路程期間に対する個人目標
  - 路程内累計との差分
- `今日の目標`
  - その日の行動目標または当日換算の目安値
  - その日の入力状況と比較しやすい形で見せる

ランキング表示候補:

- `最多アプローチ`
- `High Average アプローチ`
- `最多コミュニケーション`
- `High Average コミュニケーション`
- `最多件数`
- `High Average 件数`
- `最高支援金額`
- `High Average 金額`

比較・改善指標メモ:

- `先週比`
  - 先週同曜日比較、または直近 7 日平均との差のどちらを採用するか後で決める
- `先月比`
  - 月初から同日までの累計比較を基本候補とする
- `成長率`
  - 前期間比の増減率として表示する
- `伸びた項目`
  - アプローチ / コミュニケーション / 件数 / 金額のうち改善幅が大きいもの
- `落ちた項目`
  - 同上。責める表現ではなく、振り返りのヒントとして扱う

表示方針:

- 伸びた項目は前向きに見せる
- 落ちた項目は警告ではなく、改善余地として見せる
- 比較指標は単独表示ではなく、平均との差や目標差分と並べて意味が分かるようにする
- 月間累計と平均の両方を併記し、単純な多寡だけで評価しない

ランキング設計メモ:

- 累計ランキングと平均ランキングは分けて表示する
- 単純な多さだけでなく、継続性や質が見える平均系の指標も重視する
- メンバー同士の比較ができるが、過度に圧迫感のある見せ方は避ける
- 自分の順位が分かりやすい UI を優先する
- 管理者は個人比較だけでなく、月全体を俯瞰できることを重視する

## 12. UX 方針

- 義務感の強い業務画面にはしない
- 速度を重視する
- 操作数を減らす
- 入力完了後に少し達成感がある
- ランキングや進捗は責める印象ではなく、振り返りしやすい見せ方にする

## 12.1 プロダクト思想メモ

- 一番の課題は、現場活動するメンバーのモチベーション維持である
- 報告や実績入力のために、余計な認知負荷や作業時間を取らせない
- 入力行為そのものが負担ではなく、前向きな振り返りの入口になることを目指す
- 毎日すぐに結果が出ない日でも、努力の蓄積が数値や推移として見えることを重視する
- 単日の成果だけでなく、継続、平均、改善傾向も評価対象として見せる
- メンバーの努力が正しく実を結ぶための補助線として、このアプリを位置づける
- 便利さは目的ではなく、努力を無駄にしないための手段として扱う

- 補正履歴が追える
- 集計値に補正が反映される

### Admin Phase 5: Report App / 月次集計との整合

- 後追い追加分を月次集計へ反映
- 路程目標 / 月目標との比較値を再計算
- 必要に応じて過去報告の実績表示にも反映
- `本人入力 / 管理者補正` の内訳確認

受け入れ条件:

- `戻り` や `増額` を入れても、月全体の実績や目標差分がずれない
- 過去報告と月次集計の整合が取れる
- 管理者が「後追い込みの実績」と「元報告ベースの実績」を区別して見られる

## 14. 実装判断メモ

- Excel レイアウトの完全再現は目指さない
- Excel 由来の項目は尊重するが、UI は Web / モバイル向けに再設計する
- メンバー向け画面はモバイルファーストで設計し、`Talks` / `Testimony` の UI バランスを参照する
- ナビゲーションは初期実装からハンバーガーメニュー + ドロワー NAV を基本形にする
- メンバー画面では長いテーブルよりカードと要約表示を優先する
- メンバー入力と代表者報告を同じモデルに無理に載せない
- 途中入力と最終報告は責務を分ける
- 集計系は保存値より算出値を優先する
- 後追い実績は管理者専用操作として分離し、誰がどの理由で追加したか追跡可能にする
- 管理者画面は Excel ライクな一覧性を持たせるが、Web では読める密度を優先する
- 管理者画面は PC 優先で良いが、モバイルでも横破綻や操作不能を起こさないことを条件にする
