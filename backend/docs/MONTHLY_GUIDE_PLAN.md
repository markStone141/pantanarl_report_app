# ユニセフマンスリー多言語説明アプリ 事前メモ

## 目的

- 現場で、ユニセフのマンスリー支援について多言語で説明できるようにする
- その場で素早く開けて、言語を切り替えながら説明できることを重視する
- リアルタイム翻訳ではなく、事前に確認済みの文章を表示する

## 基本要件

- Django テンプレートベースで実装する
- 言語セレクトボックスで表示言語を切り替える
- ページは原則 1 ページ完結
- ログイン機能は不要
- 他アプリへの相互リンクは不要
- とにかく見やすさを最優先する

## 言語切り替え方針

- リアルタイム翻訳 API は使わない
- 翻訳文は事前に DB に保存する
- ただし全文はそこまで長くない想定なので、初回表示時に全言語分をまとめて読み込む方針とする
- 言語切り替え時は追加 API を叩かず、フロント側で表示だけ切り替える

### この方針の理由

- 現場での切り替えが速い
- 通信失敗の影響を受けにくい
- 翻訳文の品質を管理しやすい
- 外部翻訳 API への依存が不要

## 想定利用言語

- 日本語
- 英語
- タイ語
- ミャンマー語
- そのほか東南アジア系言語を追加できる構造

## 文字化け・表示面の考え方

- Django / PostgreSQL は UTF-8 前提なので、通常の保存と表示自体は可能
- ただし東南アジア言語はフォント依存が大きいため、実装時はフォント選定が重要
- 特にミャンマー語は字形や結合の崩れが出やすいため、実機確認を前提にする
- タイ語も行間や字間の見え方は確認する

### 実装時の注意

- UTF-8 で統一する
- 対応フォントを明示する
- iPhone / Android 実機で確認する
- 行間と余白を広めに取って視認性を優先する

## 実装方針

- 新しい Django app を 1 つ追加して、その app 内で完結させる
- 既存の accounts / reports / targets / dashboard / talks / testimony / dairymetrics とは極力疎結合にする
- URL、テンプレート、CSS、モデル、テストを新 app に閉じ込める
- 共通ナビや共通業務ロジックには依存しない
- 既存の共通 CSS への変更は原則避ける

## 既存アプリへの影響を避けるためのルール

- `backend/static/app.css` のような共通 CSS は触らない
- 共通テンプレートの変更を最小化する
- 専用テンプレートと専用 CSS を使う
- 既存アプリのモデルやビューは触らない
- テストも新 app の範囲に閉じる

## UI 方針

- 1ページ完結
- モバイルでもすぐ使えることを優先
- 言語切り替え UI はできるだけ大きく、分かりやすくする
- 説明文は大きめの文字、十分な行間、余白を取る
- 現場で疲れていても迷わないシンプルな画面にする

## 今後の実装イメージ

### サーバー側

- 初回表示時に全言語分の説明文をテンプレートへ渡す
- 必要なら簡単な管理画面で文言を更新できるようにする

### クライアント側

- 言語データをまとめて持つ
- セレクト変更で対象言語の本文へ差し替える
- 追加通信なしで切り替える

## 優先順位

1. 既存アプリへ影響を出さない構造
2. 現場での見やすさ
3. 言語切り替えの速さ
4. 翻訳文の管理しやすさ
5. 将来的な言語追加のしやすさ
